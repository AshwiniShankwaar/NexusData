# NexusData Implementation Guide

This guide provides a technical deep-dive into the NexusData architecture for engineers and automated agents.

## 1. The 5-Stage Agentic Pipeline

The core logic resides in `nexus_data/pipeline/`. Each stage is an independent agent or module:

1.  **`QueryNormalizer`**: Standardizes the User Query.
2.  **`GoalIdentifierAgent`**: Uses an LLM to extract the *intent* into a structured JSON goal (Pydantic model).
3.  **`ReferenceResolverAgent`**: Performs semantic search against the Knowledge Base (MD description + Vector embeddings) to map NL terms to valid Table/Column names.
4.  **`PlannerAgent`**: Takes the resolved goal and constructs an execution plan. It consults the **Pattern Store** (`SQLGraphStore`) to see if similar intents have been solved before.
5.  **`ExecutorAgent`**: Finalizes the SQL and runs it via SQLAlchemy.

---

## 2. Advanced Self-Healing Mechanism

One of NexusData's most powerful features is its ability to recover from database errors autonomously.

### The Flow:
1.  **Execution Failure**: The `ExecutorAgent` catches a `SQLAlchemyError`.
2.  **Traceback Analysis**: The `SelfHealer` (`nexus_data/critic/self_healer.py`) extracts the exact error message from the database (e.g., "no such column: total_rev").
3.  **LLM Correction**: The error message, the bad SQL, and the database schema are sent to the LLM with a specific "Correction Prompt".
4.  **Validation**: The corrected SQL is passed through the **Guardian** (`nexus_data/critic/guardian.py`) to ensure no destructive operations were introduced.
5.  **Retry**: The corrected SQL is executed. This loop repeats up to 3 times.

---

## 3. Knowledge Base & Memory Layers

The `KBManager` (`nexus_data/kb/manager.py`) coordinates multiple persistence layers:

-   **Metadata**: `db_info.md` (auto-generated by the Introspector).
-   **Semantic Cache**: LanceDB table storing `(embedding_vector, sql_query)`.
-   **Conversation Graph**: A SQLite-backed graph tracking the lineage of every question and answer.
-   **Audit Log**: A JSONL file recording every internal step of the pipeline for debugging.

---

## 4. Multi-DB Connection Logic

NexusData supports multiple concurrent database configurations. These are managed by the `ConfigManager` (`nexus_data/core/config_manager.py`).

### Storage:
- Connections are stored in `config.json` under the `databases` dictionary.
- Each entry contains a `DBConfig` object with the URI, exclusions, and descriptions.
- The `active_db_name` field determines which connection is loaded at startup.

### Lifecycle:
1.  **Registration**: `/add-db` triggers the `run_add_db_wizard` which validates the connection before saving.
2.  **Activation**: `/change-db` updates the `active_db_name` and re-initializes the `Orchestrator` with the new engine and a per-DB knowledge base directory (`nexus_kb/<db_name>`).
3.  **Isolation**: Each database has its own sub-directory in `nexus_kb` to prevent metadata and cache leakage between different data sources.

---

## 5. LLM Provider Mapping

To support a wide range of models (OpenAI, Google, Anthropic, OpenRouter), NexusData uses a centralized mapping in `nexus_data/core/setup_wizard.py`.

### `_MODELS_MAP`:
This dictionary defines the "Safe & Tested" models for each provider. When a user selects a provider, the UI filters the available models based on this map.

### Extension:
To add a new model or provider:
1.  Update `_MODELS_MAP` in `setup_wizard.py`.
2.  If it's a new provider, ensure the `LLMController` in `nexus_data/engine/` can handle the routing and API-specific formatting.
